Api Doc

The unitcover frontend is completly based on that api.

Clients

  • Curl based script

    • Parameters

    All path parameters are mandatory. All query parameter optional.
    • :owner - means the github owner for éxample here it is pussinboots https://github.com/pussinboots/unitcover
    • :project - means the github project for éxample here it is unitcover https://github.com/pussinboots/unitcover
    • :buildNumber - will be generated server side an has to be request with the start build method, is needed to upload test reports

    Build Resource

    offers three methods two write methods (POST) and one read (GET).

  • Start Build

    • POST http://unitcover.herokuapp.com/api/:owner/:project/builds?trigger&branch&travisBuildId

    use this method to register a new build it will return an json response contains the new buildNumber

    
{'buildNumber':45}
    Than you have to use this build number to upload test reports for them.

  • End Build

    • POST http://unitcover.herokuapp.com/api/:owner/:project/builds/:buildNumber/end

    Call this method if all test reports are uploaded it marks the build as complete and calculate some stats for it. if this method is not called than you will missing the aggregatted number test resports numbers in the build view for example.

    TestSuite Resource

  • Upload TestSuite

    • POST http://unitcover.herokuapp.com/api/:owner/:project/:buildNumber

    Request Body as xml for example you can post complete test-results.xml files like 

    <testsuite hostname="vagrant-VirtualBox" name="integration.ApplicationSpec" tests="7" errors="0" failures="0" skipped="0" time="1.975">
    <testcase name="application changed setup will work should::disable db ssl" classname="integration.ApplicationSpec" time="0.066">
        </testcase><testcase name="application changed setup will work should::enable DB logging" classname="integration.ApplicationSpec" time="0.039">
        </testcase><testcase name="application setup should should::configured with custom keystore is enabled" classname="integration.ApplicationSpec" time="0.046">
        </testcase><testcase name="application setup should should::check reditect to products.html work" classname="integration.ApplicationSpec" time="1.469">
        </testcase><testcase name="application setup should should::configured to redirect all http request to https on heroku" classname="integration.ApplicationSpec" time="0.222">
        </testcase><testcase name="application setup should should::configured with custom truststore is enabled" classname="integration.ApplicationSpec" time="0.046">
        </testcase><testcase name="application setup should should::configured with DB logging deactivate" classname="integration.ApplicationSpec" time="0.078">
    </testcase>
    <system-out></system-out>
    <system-err></system-err>
</testsuite>
    The response is json contains the testsuite ids that were created from your test suite post request before. 
    
{"testsuites":[[{"id":12},{"id":13}]]}
